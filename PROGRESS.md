- design.docx lists table names only, no columns — column design was inferred from requirements.md and the rule engine list (Phase 7) in design.docx.
- `transactions.risk_score`/`risk_level` are denormalized columns (not computed on the fly) so ledger search/filtering by risk stays index-backed per the NFR-4 performance requirement; the rule engine (Phase 7) will write to them.
- `audit_flags.rule_name` is a plain indexed string, not a DB enum, so new rule modules can be added without a migration.
- `audit_notes.cited_policy_ids`/`cited_case_ids` are JSON arrays of IDs rather than join tables — keeps the explainability citation list simple; may revisit as a proper many-to-many if we need to query "which notes cite policy X" efficiently.
- `audit_logs` has no `updated_at` and is treated as insert-only at the ORM level (NFR-2 immutability) — nothing currently enforces this at the DB level (e.g. no revoked UPDATE/DELETE grants); revisit if that becomes a compliance requirement.
- `/auth/login` accepts a JSON body (`email`/`password`) rather than OAuth2 form-encoded fields — simpler for a JSON-only API; `OAuth2PasswordBearer` is still used purely to extract the `Authorization: Bearer` header on protected routes, so Swagger's "Authorize" button still works for testing.
- `bcrypt` pinned to `4.0.1` alongside `passlib==1.7.4` — newer bcrypt (4.1+) removed an attribute passlib's version probe reads, which breaks hashing at import time.
- `require_role(*role_names)` matches role names case-insensitively (`casefold()`) since `roles.name` is admin-editable free text, not a fixed enum.
- Ledger endpoints (`GET /transactions`, `GET /transactions/{id}`) are open to any authenticated role (just `get_current_user`, no `require_role`) per the Phase 6 task — viewing the ledger isn't role-restricted, only actions like approving/flagging will be later.
- `TransactionFilterParams` is a Pydantic model bound directly as query params (FastAPI 0.115's "query parameter models": `Annotated[TransactionFilterParams, Query()]`), not individual `Query(...)` function args — keeps cross-field validation (date range, amount range) in one place via `@model_validator`, and `extra="forbid"` rejects unknown query params instead of silently ignoring them.
- `scripts/seed_transactions.py` uses a fixed `THRESHOLD_AMOUNT = 10000.00` purely as a reference point for generating "obviously too large" anomaly transactions — it does not seed `approval_limits` rows. Actual per-department/role thresholds for the Phase 7 rule engine should be seeded separately (or read from `approval_limits`) when that rule is implemented.
- Rule modules (`app/rules/*.py`) share a plain interface — `RULE_NAME: str` + `evaluate(transaction, db) -> RuleFinding | None` — rather than a class hierarchy; `app/rules/__init__.py`'s `ACTIVE_RULES` list is the only place a new rule module needs to be registered. Chosen over an ABC/plugin registry since 10 total rules (per design.docx) doesn't justify the extra machinery.
- `evaluate_transaction()` (`app/services/risk_scoring.py`) upserts `audit_flags` instead of always inserting a fresh row: an existing `OPEN` flag for a still-triggering rule gets its `risk_points`/`details` updated in place, and a flag whose rule stops triggering is marked `RESOLVED` (not deleted) — keeps `POST .../evaluate` and `.../evaluate-all` safe to call repeatedly (same lesson as the seed-script idempotency bug above) while preserving audit history instead of silently losing it.
- `threshold_violation_amount` (default $10,000) and `duplicate_detection_window_hours` (default 24h) are configurable via `Settings`/`.env`, not hardcoded in the rule modules, per the task's "configurable amount threshold" requirement. `scripts/seed_transactions.py` now reads the same `settings.threshold_violation_amount` instead of its own copy of the number, so the seed data and the rule that scores it can't drift apart.
- Risk-level score bands (`_RISK_LEVEL_BANDS` in `app/services/risk_scoring.py`: ≥75 CRITICAL, ≥50 HIGH, ≥25 MEDIUM, ≥1 LOW) are inferred, not specified in design.docx — it only gives one example (duplicate 40 + threshold 30 + vendor 15 + weekend 10 + round-number 5 = 100 → Critical). Revisited now that all 9 per-transaction rules are active (max possible 185, up from 70): left unchanged after this session's verification — they're still monotonic and produced sensible results against the seeded data (e.g. 40+35=75 correctly landed CRITICAL, 30+5+10+20=65 correctly landed HIGH), so no evidence yet that they need adjusting. Worth a deliberate re-look once real (non-synthetic) usage data exists.
- Rule engine / risk scoring endpoints (`POST /transactions/{id}/evaluate`, `POST /transactions/evaluate-all`) use `get_current_user` only, no `require_role` — matches the Phase 6 ledger endpoints' precedent (view/compute actions open to any authenticated role); revisit if the roadmap later restricts who can trigger re-scoring.
- `transactions.status` (workflow: pending/approved/rejected/flagged/cleared) and `risk_score`/`risk_level` (computed by the rule engine) are deliberately **not** coupled — `evaluate_transaction()` never touches `status`. Considered auto-setting `status=FLAGGED` when `risk_score > 0`, but that would silently overwrite transactions a human had already marked `APPROVED`/`CLEARED`; user explicitly chose to keep `status` as a purely human/workflow field. Filtering for rule-engine findings should use the `risk_level` ledger filter (added this session), not `status=flagged`.
- `scripts/seed_transactions.py`'s `clear_transactions()` also deletes `audit_flags` before deleting transactions (not just transactions) — once the Phase 7 rule engine has run, `audit_flags` rows reference transactions and a plain DELETE on `transactions` alone raises a `ForeignKeyViolation`. Safe because `audit_flags` is entirely rule-engine-derived output (regenerated by re-running `evaluate-all`); `audit_cases`/`audit_notes` are still deliberately left alone.
- Point values for the 5 rules design.docx doesn't weight in its example (debit_credit_mismatch=10, missing_approval=20, inactive_vendor=20, split_payment=35) are inferred, not specified — design.docx's "Total = 100" example only weights duplicate/threshold/vendor/weekend/round-number. Reasoning: split_payment (35) is weighted close to duplicate/threshold since deliberate structuring to dodge an approval control is as serious as the threshold violation it's evading; missing_approval/inactive_vendor (20 each) are real control gaps, weighted above the "soft" signals (round_number=5, weekend_holiday=10) but below outright evasion; debit_credit_mismatch (10) is the weakest signal here since a CREDIT against a vendor could be a legitimate reversal, not just miscoding.
- `benfords_law.py` is **not** a per-transaction rule and is deliberately excluded from `ACTIVE_RULES` — Benford's Law is a statistical property of a whole sample's leading-digit distribution, so no single transaction can "violate" it the way it can be over the threshold or a duplicate. It's run once per `evaluate-all` call (chi-square test, 8 degrees of freedom, p=0.05 critical value ≈15.507) over the whole evaluated batch and returned as a separate `benfords_law` field, never written to any transaction's `audit_flags`/`risk_score`. Note for future sessions: the synthetic seed data is generated via `random.randint()` (uniform), which does **not** naturally follow Benford's Law the way real transaction amounts do (real amounts follow it due to multiplicative/log-scale pricing processes) — so `evaluate-all` against the seed data will always report `is_significant: true`. That's expected and correct given how the data is generated, not a sign the seed data or the check is broken; it just means this particular check can't be meaningfully sanity-tested against synthetic uniform-random amounts.

- **Phase 10:** Structured output on Groq is enforced via forced tool-calling (`tool_choice` pinned to a single `submit_audit_note` function whose `parameters` is `AuditNoteDraft.model_json_schema()`), not response-format JSON mode — Groq's `{"type": "json_object"}` mode only guarantees syntactically valid JSON, not conformance to a specific schema, whereas forcing a tool call with a JSON-Schema `parameters` definition gets much closer to Claude's `output_config.format` guarantee. The response is still explicitly re-validated with `AuditNoteDraft.model_validate()` (catching both `json.JSONDecodeError` and pydantic `ValidationError` as a single `generation_failed` error) rather than trusted blindly, since Groq's schema adherence is not a hard guarantee the way Claude's structured outputs are.
- **Phase 10:** `audit_notes.audit_flag_id` is a required FK to exactly one flag, but design.docx's workflow drafts one holistic audit note per *transaction* (which can have several open flags). Resolved by anchoring the note to the highest-`risk_points` open flag (ties broken by lowest id) as the "primary reason" (`_pick_primary_flag()` in `audit_note_service.py`), while the prompt and generated content still address every open flag on the transaction, not just the primary one.
- **Phase 10:** `generate_audit_note()` defensively filters the LLM's `cited_policy_ids` down to only the policy_ids actually present in the retrieved RAG context before persisting — never trusts the model not to hallucinate a citation to a policy that was never shown to it. This is load-bearing for the explainability NFR (docs/requirements.md NFR-3: a reviewer must be able to verify the citation, which is impossible for a policy_id that was never part of the prompt).
- **Phase 11:** Dashboard's third stat card is **"Cleared Transactions"** (count of `status=cleared`, meta = count of `status=pending` as "pending review"), not the design handoff's literal **"Notes Drafted"**. The backend has no GET/list/count endpoint for `audit_notes` (only `POST /transactions/{id}/generate-audit-note`), and the task required every stat card to be wired to real API data, not hardcoded — user chose this relabeling (over a static placeholder, dropping to 2 cards, or pausing frontend work to add a backend endpoint first) to keep the 3-card layout with every number honest. ~~Revisit once a real audit-notes count endpoint exists~~ — **investigated and resolved 2026-07-21: staying with "Cleared Transactions," not switching.** Now that `GET /audit-notes` exists (2026-07-20), a literal switch was queried against live data: `status=draft` returns 6 rows, but those 6 rows resolve to only **3 distinct transactions** (1485, 1613, 1436), because each already has a duplicate note row from the known unfixed `generate_audit_note()` idempotency bug (see Blockers below) — both duplicates of all three pairs currently sit at `draft`. A raw `COUNT(status=draft)` "Notes Drafted" card would silently show 6 when only 3 transactions actually need first-look attention; making it accurate would require `COUNT(DISTINCT transaction_id)` server-side, not a row count, and there's no endpoint that does that today. Meanwhile "Cleared Transactions" (currently 93) has no known data-quality issue — it reads `transactions.status` directly, nothing duplicates or double-counts it. Decision: keep "Cleared Transactions" as-is; **this also reinforces fixing the `generate_audit_note()` idempotency bug as a real priority**, since it's now shown to skew more than just orphaned DB rows — it would have corrupted a user-facing Dashboard stat too, had this switch been made without checking.
- **Phase 11:** `GET /api/v1/transactions` list items only carry `vendor_id`/`department_id` (no name strings — see `TransactionOut` in `backend/app/schemas/transaction.py`), but the Dashboard table needs vendor/department *names*. There's no vendors/departments list endpoint either. Resolved by fetching `GET /transactions/{id}` (which nests `vendor`/`department` summaries) per row for just the 8 rows actually rendered in "Recent Flagged Transactions" (`useQueries` in `frontend/src/pages/DashboardPage.tsx`) rather than adding a backend endpoint — acceptable N+1 at this scale (8 rows), revisit if the table grows a lot or gets pagination.
- **Phase 11:** "Avg Risk Score" is averaged over the fetched high+critical (flagged) transactions, not literally every transaction in the ledger — there's no aggregate/stats endpoint, and `risk_level=high`/`risk_level=critical` are each fetched with `page_size=100` (the API's max), which covers the full flagged population unless either bucket exceeds 100 rows. "Flagged Transactions" and "Cleared Transactions" card values use the API's own `total` field instead (exact regardless of `page_size`), so only the average is a bounded sample.
- **Phase 11:** Tailwind **v3** (`darkMode: 'class'`, `tailwind.config.js` `theme.extend.colors`), not v4, despite v4 installing by default via `npm install tailwindcss` at the time of this session — v3's config-file + `dark:` variant pattern was a better fit for a token-heavy dual-theme design (many one-off `rgba()` risk-badge colors per light/dark pair) than v4's newer CSS-first `@theme`/`@custom-variant` approach, and is far better documented for this exact use case.
- **Phase 11:** `frontend/.env`/`.env.example` holds `VITE_API_BASE_URL` (defaults to `http://localhost:8000/api/v1`) — the only frontend env var so far; `.env` is gitignored, `.env.example` is committed.
- **Phase 11:** Ledger Explorer's vendor/department filter dropdowns and table-cell name lookups (`frontend/src/lib/filterOptions.ts`) are a hardcoded id→name list, not a live API call — there's still no vendors/departments list endpoint (same gap Dashboard hit first, see the `Phase 11` entry above about `TransactionOut` only having ids). Deliberately **not** reusing Dashboard's per-row `GET /transactions/{id}` detail-fetch pattern here: that's an acceptable N+1 at Dashboard's fixed 8-row scale, but Ledger's page_size goes up to 100, where 100 parallel detail-fetches per page load would be real overhead for zero benefit (the names are static seed data, not something that changes per transaction). The hardcoded list was sourced directly from the live seeded Postgres DB (`docker exec ... psql`), not the seed script's insertion order, so it reflects actual DB ids. Will go stale if vendors/departments are ever added/renamed outside `scripts/seed_transactions.py`'s current fixed set — revisit alongside the future Admin Panel if vendor/department management becomes editable.
- **Phase 11 (Reporting Service):** `services/report_service.py`'s "flagged/audited transactions" report is scoped to `risk_level IN (medium, high, critical)`, **not** "any transaction with at least one `audit_flags` row" — decided from a direct query against the live seeded DB, not guessed. Of the 159 transactions with any `audit_flags` row (open or resolved), **116 (73%) currently sit at `risk_level=LOW`** — typically a single round-number or weekend-holiday flag, i.e. already-resolved or low-signal history the rule engine's own denormalized `risk_score`/`risk_level` no longer weights as meaningful — versus only **43** genuinely at `medium`/`high`/`critical`. `risk_level` is recomputed fresh on every `evaluate_transaction()`/`evaluate-all` call (see the Phase 7 entry above: an existing flag whose rule stops triggering is marked `RESOLVED`, not deleted, and `risk_score` is resummed from scratch), so it already reflects *current* risk — unlike `audit_flags` existence, which is permanent history that never un-happens even after the underlying issue is fixed. Using flag-existence instead would have made the report ~3.7x larger and mostly noise. `REPORTABLE_RISK_LEVELS` in `report_service.py` is the one place this threshold is defined; revisit only if a future rule intentionally wants LOW-risk transactions surfaced in reports (none currently do).
- **Phase 11 (Admin Panel):** `PATCH /vendors/{id}` is gated with `require_role("Admin")` alone — the first *single*-role gate anywhere in the app (every prior write, submit/approve/reject, was gated to a *pair* of roles, e.g. `require_role("Auditor", "Admin")`). No mechanism changes were needed; `require_role(*allowed_roles)` already supported an arbitrary number of role names, this is just the first caller passing exactly one. Verified live that Auditor and Finance Manager both correctly get a clean `403`, not a `500` or a silent pass-through.
- **Phase 11 (Admin Panel):** `GET /rules` has no service-layer function, unlike every other endpoint added so far — it directly imports `ACTIVE_RULES` from `app/rules/__init__.py` in the route handler. The additive service-layer pattern exists specifically so HTTP routes and MCP tools can share logic without duplicating a DB query; `ACTIVE_RULES` is a static Python list, not a DB query, and there's no MCP tool consuming it, so a service function here would be an empty pass-through wrapping nothing. Revisit only if an MCP tool or a second HTTP caller for the same data appears.
- **Phase 11 (Admin Panel): `GET/POST /users` and department create/edit/deactivate are explicitly out of scope for this pass** — only vendor management (`GET /vendors`, `PATCH /vendors/{id}`) and the rule overview (`GET /rules`) were built. The 2026-07-19 investigation confirmed no user-management endpoint exists at all (only `POST /auth/login`/`GET /auth/me`), which also means there's currently no cheap way to get a live user count for an Admin Panel overview — vendors/departments can reuse the `page_size=1`-just-read-`total` trick once list endpoints exist for them, but users have no list endpoint to piggyback on. Deferred alongside the other known gaps (see the consolidated "Deferred / Known Gaps" section below) rather than built speculatively ahead of an actual Admin Panel user-management need.
