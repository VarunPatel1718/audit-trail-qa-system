## Key decisions log
_(Record anything you decided that deviates from the original design doc, so future-you knows why)_

- design.docx lists table names only, no columns — column design was inferred from requirements.md and the rule engine list (Phase 7) in design.docx.
- `transactions.risk_score`/`risk_level` are denormalized columns (not computed on the fly) so ledger search/filtering by risk stays index-backed per the NFR-4 performance requirement; the rule engine (Phase 7) will write to them.
- `audit_flags.rule_name` is a plain indexed string, not a DB enum, so new rule modules can be added without a migration.
- `audit_notes.cited_policy_ids`/`cited_case_ids` are JSON arrays of IDs rather than join tables — keeps the explainability citation list simple; may revisit as a proper many-to-many if we need to query "which notes cite policy X" efficiently.
- `audit_logs` has no `updated_at` and is treated as insert-only at the ORM level (NFR-2 immutability) — nothing currently enforces this at the DB level (e.g. no revoked UPDATE/DELETE grants); revisit if that becomes a compliance requirement.
- `/auth/login` accepts a JSON body (`email`/`password`) rather than OAuth2 form-encoded fields — simpler for a JSON-only API; `OAuth2PasswordBearer` is still used purely to extract the `Authorization: Bearer` header on protected routes, so Swagger's "Authorize" button still works for testing.
- `bcrypt` pinned to `4.0.1` alongside `passlib==1.7.4` — newer bcrypt (4.1+) removed an attribute passlib's version probe reads, which breaks hashing at import time.
- `require_role(*role_names)` matches role names case-insensitively (`casefold()`) since `roles.name` is admin-editable free text, not a fixed enum.
- Ledger endpoints (`GET /transactions`, `GET /transactions/{id}`) are open to any authenticated role (just `get_current_user`, no `require_role`) per the Phase 6 task — viewing the ledger isn't role-restricted, only actions like approving/flagging will be later.
- `TransactionFilterParams` is a Pydantic model bound directly as query params (FastAPI 0.115's "query parameter models": `Annotated[TransactionFilterParams, Query()]`), not individual `Query(...)` function args — keeps cross-field validation (date range, amount range) in one place via `@model_validator`, and `extra="forbid"` rejects unknown query params instead of silently ignoring them.
- `scripts/seed_transactions.py` uses a fixed `THRESHOLD_AMOUNT = 10000.00` purely as a reference point for generating "obviously too large" anomaly transactions — it does not seed `approval_limits` rows. Actual per-department/role thresholds for the Phase 7 rule engine should be seeded separately (or read from `approval_limits`) when that rule is implemented.
- Rule modules (`app/rules/*.py`) share a plain interface — `RULE_NAME: str` + `evaluate(transaction, db) -> RuleFinding | None` — rather than a class hierarchy; `app/rules/__init__.py`'s `ACTIVE_RULES` list is the only place a new rule module needs to be registered. Chosen over an ABC/plugin registry since 10 total rules (per design.docx) doesn't justify the extra machinery.
- `evaluate_transaction()` (`app/services/risk_scoring.py`) upserts `audit_flags` instead of always inserting a fresh row: an existing `OPEN` flag for a still-triggering rule gets its `risk_points`/`details` updated in place, and a flag whose rule stops triggering is marked `RESOLVED` (not deleted) — keeps `POST .../evaluate` and `.../evaluate-all` safe to call repeatedly (same lesson as the seed-script idempotency bug above) while preserving audit history instead of silently losing it.
- `threshold_violation_amount` (default $10,000) and `duplicate_detection_window_hours` (default 24h) are configurable via `Settings`/`.env`, not hardcoded in the rule modules, per the task's "configurable amount threshold" requirement. `scripts/seed_transactions.py` now reads the same `settings.threshold_violation_amount` instead of its own copy of the number, so the seed data and the rule that scores it can't drift apart.
- Risk-level score bands (`_RISK_LEVEL_BANDS` in `app/services/risk_scoring.py`: ≥75 CRITICAL, ≥50 HIGH, ≥25 MEDIUM, ≥1 LOW) are inferred, not specified in design.docx — it only gives one example (duplicate 40 + threshold 30 + vendor 15 + weekend 10 + round-number 5 = 100 → Critical). Revisited now that all 9 per-transaction rules are active (max possible 185, up from 70): left unchanged after this session's verification — they're still monotonic and produced sensible results against the seeded data (e.g. 40+35=75 correctly landed CRITICAL, 30+5+10+20=65 correctly landed HIGH), so no evidence yet that they need adjusting. Worth a deliberate re-look once real (non-synthetic) usage data exists.
- Rule engine / risk scoring endpoints (`POST /transactions/{id}/evaluate`, `POST /transactions/evaluate-all`) use `get_current_user` only, no `require_role` — matches the Phase 6 ledger endpoints' precedent (view/compute actions open to any authenticated role); revisit if the roadmap later restricts who can trigger re-scoring.
- `transactions.status` (workflow: pending/approved/rejected/flagged/cleared) and `risk_score`/`risk_level` (computed by the rule engine) are deliberately **not** coupled — `evaluate_transaction()` never touches `status`. Considered auto-setting `status=FLAGGED` when `risk_score > 0`, but that would silently overwrite transactions a human had already marked `APPROVED`/`CLEARED`; user explicitly chose to keep `status` as a purely human/workflow field. Filtering for rule-engine findings should use the `risk_level` ledger filter (added this session), not `status=flagged`.
- `scripts/seed_transactions.py`'s `clear_transactions()` also deletes `audit_flags` before deleting transactions (not just transactions) — once the Phase 7 rule engine has run, `audit_flags` rows reference transactions and a plain DELETE on `transactions` alone raises a `ForeignKeyViolation`. Safe because `audit_flags` is entirely rule-engine-derived output (regenerated by re-running `evaluate-all`); `audit_cases`/`audit_notes` are still deliberately left alone.
- Point values for the 5 rules design.docx doesn't weight in its example (debit_credit_mismatch=10, missing_approval=20, inactive_vendor=20, split_payment=35) are inferred, not specified — design.docx's "Total = 100" example only weights duplicate/threshold/vendor/weekend/round-number. Reasoning: split_payment (35) is weighted close to duplicate/threshold since deliberate structuring to dodge an approval control is as serious as the threshold violation it's evading; missing_approval/inactive_vendor (20 each) are real control gaps, weighted above the "soft" signals (round_number=5, weekend_holiday=10) but below outright evasion; debit_credit_mismatch (10) is the weakest signal here since a CREDIT against a vendor could be a legitimate reversal, not just miscoding.
- `benfords_law.py` is **not** a per-transaction rule and is deliberately excluded from `ACTIVE_RULES` — Benford's Law is a statistical property of a whole sample's leading-digit distribution, so no single transaction can "violate" it the way it can be over the threshold or a duplicate. It's run once per `evaluate-all` call (chi-square test, 8 degrees of freedom, p=0.05 critical value ≈15.507) over the whole evaluated batch and returned as a separate `benfords_law` field, never written to any transaction's `audit_flags`/`risk_score`. Note for future sessions: the synthetic seed data is generated via `random.randint()` (uniform), which does **not** naturally follow Benford's Law the way real transaction amounts do (real amounts follow it due to multiplicative/log-scale pricing processes) — so `evaluate-all` against the seed data will always report `is_significant: true`. That's expected and correct given how the data is generated, not a sign the seed data or the check is broken; it just means this particular check can't be meaningfully sanity-tested against synthetic uniform-random amounts.
- `scripts/seed_transactions.py` baseline transactions were changed from uniform-random `debit_credit`/`account_number` to realistic-and-clean (`debit_credit=DEBIT` always, `account_number` always equal to the vendor's `bank_account_number`) once the debit_credit_mismatch and vendor_account_mismatch rules existed — otherwise ~50%/100% of baseline data would false-positive on those rules respectively, making "sanity-check a few flagged results per rule" impossible to do meaningfully. Anomaly blocks override exactly the field each rule checks.
- Fixed a gap in this file's own Phase 7 checklist: "Debit/credit validation" was missing from the original list (design.docx lists it as one of the 10 rule modules) — added it back rather than leaving it undocumented.
- **Phase 8:** `docs/design.docx` and `docs/roadmap.docx` are plain UTF-8 text saved with a `.docx` extension, not real Office Open XML — `python-docx`/binary tooling fails on them (`PackageNotFoundError`). Read them as plain text instead. Worth renaming to `.txt` in a future session, but left as-is since the task only needed their content, not the extension fixed.
- **Phase 8:** Policy chunks are split by real chapter/clause numbering (e.g. "4.1.5"), not fixed-size windows, per the task requirement — `app/services/policy_chunking.py`. Footnote/citation text is deliberately dropped rather than kept (filtered out by point size, <9.5pt vs 10-12pt body text), since it's supplementary citation material, not operative policy text, and its numbered-footnote lines would otherwise be indistinguishable from real numbered clauses.
- **Phase 8:** `Policy.clause_ref`/`Policy.chapter` are new plain-string columns (migration `4ec0911d45e2`), unindexed beyond a plain btree index — mirrors the existing `audit_flags.rule_name` precedent of "plain indexed string, not an enum" so new documents with different numbering conventions don't need a migration.
- **Phase 8:** Chose **BAAI/bge-base-en-v1.5** over the task's suggested `bge-small` after verification showed `bge-small` ranked the correct answer clause for "what is the reporting threshold for fraud in NBFCs" at 19th out of 20 candidates. The task's wording ("e.g. BAAI/bge-small-en-v1.5") treats the model as a suggestion, not a hard requirement, and retrieval correctness on the task's own verification queries took priority. `bge-base` is still fully local/free via `sentence-transformers`, just larger (768-dim vs 384-dim).
- **Phase 8:** Embedding input is enriched with ancestor-clause headings and the chapter label (`_build_embedding_texts()` in `scripts/ingest_policies.py`), separate from the stored/returned `content` — a deeply nested clause is often a terse operative sentence with none of its section's topical vocabulary (e.g. clause 6.2.1 never uses the word "report"; its parent heading 6.2 is literally "Modalities of *Reporting* Incidents of Fraud to RBI"). This is a general context-enrichment technique validated against a second, differently-phrased query and a third query against an unrelated document (KYC), not a fix tailored to one query's exact wording.
- **Phase 8:** `policy_search_default_top_k` is 8, not 5 — chosen after verification showed correct-but-non-lexically-matching clauses (like 6.2.1 above) can land just outside a top-5 cutoff even with a good embedding model; 8 comfortably covers the task's verification queries while still being a reasonable default result-list size. `PolicySearchRequest.top_k` remains caller-overridable, 1-20.
- **Phase 8:** `ingest_policies.py` fully clears and rebuilds both `policies` (Postgres) and the Qdrant `policies` collection on every run rather than diffing — matches the existing `seed_transactions.py`/rule-engine precedent of "safe to re-run" scripts, and 322 chunks re-embed in well under a minute locally so there's no cost pressure to make it incremental.
- **Phase 9:** `starlette==0.38.6`/`sse-starlette==2.1.3` are now pinned in `requirements.txt` alongside `mcp==1.28.1` — installing plain `mcp` (or `mcp[cli]`) without a pin resolves to `starlette>=0.49`, which breaks `fastapi==0.115.0` at import time (`Router.__init__() got an unexpected keyword argument 'on_startup'`). If `fastapi` is ever upgraded past the `0.115.x` line, revisit whether this pin is still needed.

Revisit if the MCP server is ever exposed over a network transport rather than run locally via stdio.

Revisit only if manual flags are later given real point values with `evaluate_transaction()` also updated to preserve them.
- **Phase 9:** Extracted `app/services/ledger_service.py` (`search_transactions`, `get_transaction_by_id`) out of `app/api/transactions.py`, and moved `app/api/rule_engine.py`'s private `_open_flags` into `app/services/risk_scoring.py` as `get_open_flags` — both were previously inlined in route handlers with no way for the new MCP tools to reuse them without copy-pasting. The HTTP routes now just call these same functions; behavior is unchanged (re-verified end-to-end after the refactor).

Also noted in `docs/requirements.md` FR6.
